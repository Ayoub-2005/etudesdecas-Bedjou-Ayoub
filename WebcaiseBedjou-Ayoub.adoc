=== Question 1.2 – Audit de sécurité

==== Analyse de la procédure actuelle

Le cahier des charges indique que l'identifiant et le mot de passe
seront transmis au client par courrier électronique.

Cette pratique présente plusieurs risques de sécurité.

==== Analyse selon les critères DICP

*Confidentialité*

Le courrier électronique n'est pas un moyen de communication sécurisé.
Le mot de passe peut être intercepté lors de la transmission ou consulté
par une personne ayant accès à la boîte mail du client.
La confidentialité des données d'authentification n'est donc pas garantie.

*Intégrité*

Un message électronique peut être intercepté ou altéré.
Une personne malveillante pourrait modifier ou utiliser les identifiants
avant le client.

*Preuve (traçabilité)*

Si plusieurs personnes ont accès au mot de passe transmis par email,
il devient impossible de garantir l'identité réelle de l'utilisateur
connecté au système.

==== Non conformité avec le RGPD

Le RGPD impose la mise en place de mesures techniques
permettant d'assurer la sécurité des données personnelles.

Le mot de passe constitue une donnée d'authentification sensible.
Son envoi en clair par courrier électronique ne garantit pas
sa protection contre l'accès non autorisé.

==== Proposition d'un processus d'inscription sécurisé

Un processus d'activation de compte sécurisé peut être mis en place.

1. Le client remplit un formulaire d'inscription sur le site.
2. Le système crée un compte utilisateur inactif.
3. Un lien d'activation contenant un jeton unique est envoyé par email.
4. Le client clique sur ce lien et choisit son mot de passe.
5. Le mot de passe est stocké dans la base de données sous forme hachée.

Ce mécanisme permet d'éviter la transmission de mots de passe en clair
et améliore la sécurité du système.

=== Question 1.3 – Analyse de la table proposée

La table proposée possède la structure suivante :

FormuleSouscrite(idPointDeVente, idFormule)
Clé primaire : (idPointDeVente, idFormule)

Cette structure ne permet pas de gérer correctement l’historique
des formules souscrites pour plusieurs raisons.

Tout d’abord, la table ne contient aucune information temporelle
comme une date de début ou une date de fin d’activation de la formule.
Il est donc impossible de connaître la durée pendant laquelle
une formule a été utilisée par un point de vente.

Or le montant à facturer dépend de la durée d’activation
de chaque formule.

Ensuite, la clé primaire composée de (idPointDeVente, idFormule)
empêche d’enregistrer plusieurs fois la même formule
pour un même point de vente.

Par exemple, si un client utilise une formule "Silver",
passe ensuite à la formule "Gold", puis revient à "Silver",
la base de données ne pourra pas enregistrer ce nouvel abonnement.

Enfin, l’absence de date ne permet pas de vérifier la règle
selon laquelle un changement de formule ne peut être effectué
qu’une seule fois par jour.

La structure proposée ne permet donc pas de conserver l’historique
nécessaire à la facturation mensuelle.

=== Question 1.4 – Correction de la structure de la table

Afin de conserver l’historique des formules souscrites,
il est nécessaire d'ajouter des informations temporelles
dans la table.

Chaque souscription doit contenir la date de début
et éventuellement la date de fin d’activation de la formule.

Il est également préférable d’ajouter un identifiant
unique pour chaque souscription afin de pouvoir enregistrer
plusieurs changements de formule pour un même point de vente.

La structure corrigée de la table peut être la suivante :

FORMULE_SOUSCRITE

- idSouscription (PK)
- idPointDeVente (FK)
- idFormule (FK)
- dateDebut
- dateFin

Cette structure permet de conserver l’historique des formules
utilisées par chaque point de vente et de calculer la durée
d’activation nécessaire à la facturation mensuelle.

=== Question 1.5 – Déclencheur d’audit

Afin de garantir la traçabilité des changements de formule,
il est possible de mettre en place un déclencheur (trigger)
qui enregistre chaque modification dans une table d’audit.

Ce déclencheur se déclenche après la modification d’une ligne
dans la table FORMULE_SOUSCRITE.

Lorsque la formule associée à un point de vente est modifiée,
le trigger enregistre l’ancienne et la nouvelle formule
dans une table d’audit.

[source,sql]
----
CREATE TRIGGER tg_audit_changement_formule
AFTER UPDATE ON FORMULE_SOUSCRITE
FOR EACH ROW
BEGIN

    IF OLD.idFormule <> NEW.idFormule THEN

        INSERT INTO AUDIT_CHANGEMENT_FORMULE(
            idPointDeVente,
            ancienneFormule,
            nouvelleFormule,
            dateChangement
        )
        VALUES(
            OLD.idPointDeVente,
            OLD.idFormule,
            NEW.idFormule,
            CURRENT_TIMESTAMP
        );

    END IF;

END;
----

SELECT DISTINCT nom, prenom, mail

=== Question 2.2 – Analyse de vulnérabilité

La variable seuilVentes provient directement de la saisie de l’utilisateur
dans l’interface JavaFX et est utilisée dans la construction de la requête SQL
par concaténation de chaînes de caractères.

Cette pratique expose l’application à une vulnérabilité de type
*Injection SQL*, qui fait partie des failles majeures du classement OWASP.

En effet, un utilisateur malveillant pourrait saisir une valeur contenant
du code SQL afin de modifier le comportement de la requête.
Par exemple, la saisie d’une valeur telle que :

0 OR 1=1

rendrait la condition de la clause HAVING toujours vraie et pourrait permettre
d’accéder à des données non autorisées.

Dans des cas plus graves, un attaquant pourrait tenter d’exécuter
des commandes SQL supplémentaires afin de modifier ou supprimer
des données de la base.

Cette vulnérabilité est due à l’utilisation de la concaténation de chaînes
pour construire la requête SQL.

Les bonnes pratiques de sécurité recommandées par l’OWASP consistent
à utiliser des requêtes paramétrées (PreparedStatement) afin de séparer
le code SQL des données fournies par l’utilisateur et éviter ainsi
les attaques par injection SQL.

=== Question 2.3 – Sécurisation de l’accès à la base avec PreparedStatement

Afin d’éviter les attaques par injection SQL, la requête doit être
écrite en utilisant une requête paramétrée avec l’interface
PreparedStatement de JDBC.

La valeur saisie par l’utilisateur n’est plus concaténée directement
dans la requête SQL mais remplacée par un paramètre "?".
La valeur est ensuite transmise séparément via la méthode setInt().

[source,sql]
----
SELECT nom, prenom, tel, mail
FROM Conso
JOIN Vente ON idConso = Conso.id
GROUP BY Conso.id
HAVING COUNT(*) > ?
----

[source,java]
----
String requete =
"SELECT nom, prenom, tel, mail " +
"FROM Conso " +
"JOIN Vente ON idConso = Conso.id " +
"GROUP BY Conso.id " +
"HAVING COUNT(*) > ?";

PreparedStatement pstmt = dbConnect.prepareStatement(requete);

pstmt.setInt(1, seuilVentes);

ResultSet res = pstmt.executeQuery();
----
